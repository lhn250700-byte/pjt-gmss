# 게시글 로딩 속도 개선 가이드

게시판 목록·상세·인기글 로딩이 느릴 때 **서버(백엔드)에서 불러오는 방식**을 어떻게 바꾸면 좋은지 정리했습니다.

---

## 1. 이미 적용된 개선 사항

### 1.1 댓글 N+1 제거 (백엔드)

- **위치:** `BbsCommentRepository.findByBbsIdAndDelYnOrderByCreatedAtAsc`
- **내용:** `@EntityGraph(attributePaths = {"memberId"})` 추가로, 댓글 목록 조회 시 **회원 정보를 한 번에 조인**해 가져옵니다.
- **효과:** 게시글 상세 진입 시 댓글 100개면 1번 쿼리로 처리 (기존엔 1 + 100번 쿼리 가능).

### 1.2 인기글 캐시 (백엔드)

- **위치:** `BbsService.findRealtimePopularPosts()`, `findWeeklyPopularPosts()`
- **내용:** 실시간/주간 인기글 결과를 **90초간 메모리 캐시**. 동일 탭 재요청 시 DB 조회 생략.
- **효과:** 인기글 탭 전환·새로고침 시 체감 속도 향상.

### 1.3 응답 압축 (백엔드)

- **위치:** `application.properties`
- **내용:** `server.compression.enabled=true` 로 JSON 등 텍스트 응답 gzip 압축.
- **효과:** 전송 데이터 크기 감소 → 네트워크 구간에서 체감 속도 개선.

### 1.4 프론트 캐시 (이미 있음)

- **위치:** `BoardList.jsx`
- **내용:** 목록 60초, 인기글 90초 캐시로 재요청 시 API 생략.
- **효과:** 같은 페이지 재방문 시 즉시 표시.

---

## 2. 추가로 적용할 수 있는 방법 (서버에서 불러오는 방식 변경)

### 2.1 목록 API에서 본문(content) 제외

**문제:** 목록 조회 시에도 `content` 필드가 포함되어 응답이 커짐.

**방법:**

- 백엔드에 **목록 전용 DTO**를 두고, `bbsId`, `title`, `bbs_div`, `created_at`, `views`, 작성자(닉네임), 댓글 수·좋아요 수만 반환하고 **`content`는 넣지 않음**.
- Repository에 `SELECT` 대상 컬럼을 제한한 쿼리 또는 DTO 프로젝션 쿼리 추가.
- 프론트는 기존 목록 API 대신 이 API를 호출하도록 변경.

**효과:** 목록 한 페이지당 전송량 감소 → 목록 로딩이 빨라짐.

---

### 2.2 게시글 상세 “한 번에” API

**문제:** 상세 화면에서 게시글 조회 / 댓글 목록 / 좋아요 수를 **3번 요청** (Promise.all로 병렬이지만 왕복 3번).

**방법:**

- 백엔드에 **단일 API** 추가. 예: `GET /api/bbs/{id}/detail`
- 이 API에서 **게시글 + 댓글 목록 + 좋아요/싫어요 수**를 한 번에 조회해 하나의 JSON으로 반환.
- 프론트는 기존 `getById` + `getComments` + `getLikeCounts` 대신 이 API 1번만 호출.

**효과:** 상세 페이지 진입 시 요청 1번으로 정리 → 지연 시간 감소.

---

### 2.3 인기글 쿼리 최적화 (DB에서 10건만)

**현재:** 최근 200건을 DB에서 가져온 뒤, Java에서 점수 계산·정렬해 상위 10건만 사용.

**방법:**

- 인기 점수를 **DB 쿼리로 계산** (서브쿼리·윈도우 함수 등)하고, `ORDER BY 점수 DESC LIMIT 10` 으로 **10건만** 조회.
- 또는 인기글 전용 **마테리얼라이즈드 뷰 / 주기적 집계 테이블**을 두고, 그 테이블만 조회.

**효과:** 인기글 API의 DB 부하 및 데이터 전송량 감소.

---

### 2.4 DB 인덱스

아래 컬럼 조합에 인덱스가 있으면 목록·상세·댓글 조회가 빨라집니다.

- `bbs(del_yn, created_at)` — 목록 최신순
- `bbs(bbs_div, del_yn, created_at)` — 분류별 목록
- `bbs_comment(bbs_id, del_yn)` — 댓글 목록
- `bbs_like(bbs_id)` — 좋아요 수 집계

이미 있다면 유지, 없다면 DBA 또는 팀 규칙에 맞춰 추가.

---

### 2.5 프론트 추가 개선 (참고)

- **목록:** 처음에는 1페이지만 요청하고, 스크롤 시 다음 페이지 로딩(무한 스크롤)으로 전환하면 첫 화면 로딩이 가벼워짐.
- **상세:** 게시글 본문만 먼저 보여주고, 댓글은 “댓글 더보기” 또는 스크롤 시 로딩하면 첫 페인트가 빨라짐 (이미 “한 번에 API”를 쓰면 1요청으로 처리 가능).

---

## 3. 요약

| 구분 | 적용 여부 | 설명 |
|------|-----------|------|
| 댓글 N+1 제거 | ✅ 적용됨 | EntityGraph로 댓글+회원 한 번에 조회 |
| 인기글 캐시 | ✅ 적용됨 | 90초 메모리 캐시 |
| 응답 gzip | ✅ 적용됨 | application.properties |
| 목록에서 content 제외 | 선택 | 목록 전용 DTO·쿼리로 전송량 감소 |
| 상세 1회 API | 선택 | 상세+댓글+좋아요 한 번에 응답 |
| 인기글 DB 10건만 | 선택 | 쿼리/집계 테이블로 최적화 |
| 인덱스 | 확인 권장 | bbs, bbs_comment, bbs_like 등 |

**서버에서 불러오는 방식**을 바꾸려면 위 “2. 추가로 적용할 수 있는 방법” 순서대로 검토하면 됩니다. 목록이 느리면 2.1, 상세가 느리면 2.2를 우선 적용하는 것을 권장합니다.
